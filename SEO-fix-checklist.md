# SEO 修复清单

这是一份面向当前站点的技术 SEO 修复清单，前提是 **不改文案、不改品牌、不改 URL**。文档主要覆盖 `canonical`、索引治理、结构化数据、`sitemap`、性能、内部链接等问题，作为后续技术优化的长期参考资料。

## 使用范围与约束

- 不修改页面正文文案
- 不修改品牌名称
- 不修改现有 URL 结构
- 不扩展到外链、PR、内容运营层面
- 仅聚焦当前已识别的技术 SEO 与结构 SEO 修复项

## P0-立刻执行

### 1. 首页 canonical 自引用补齐

- **问题**
  - 首页缺少明确的自引用 canonical，首页规范化信号不足。
- **涉及文件**
  - `src/app/layout.tsx`
  - 如需单独拆首页 metadata，则涉及首页对应 metadata 文件
- **修复动作**
  - 为首页输出明确的 canonical，指向 `https://rgbatohex.com/`。
  - 确保首页 canonical 不受参数、协议、尾斜杠等情况影响。
- **预期收益**
  - 强化首页规范 URL 信号。
  - 降低首页出现重复规范化歧义的风险。

### 2. 博客文章 canonical 修复

- **问题**
  - 博客 layout 当前将 canonical 固定指向 `/blog`，文章页可能错误继承，导致文章页信号被折叠到博客首页。
- **涉及文件**
  - `src/app/blog/layout.tsx`
  - `src/app/blog/*/page.tsx`
- **修复动作**
  - 将博客首页的 canonical 与博客文章页的 canonical 分离处理。
  - 博客首页保留指向 `/blog`。
  - 每篇文章页输出自引用 canonical，指向自己的文章 URL。
- **预期收益**
  - 避免文章页索引信号丢失。
  - 提升博客文章页的独立收录与排名能力。

### 3. 博客文章双 H1 清理

- **问题**
  - 部分博客文章同时存在隐藏 SEO 块 H1 和正文主 H1，造成同页多个 H1。
- **涉及文件**
  - `src/app/blog/rgba-to-hex-converter/page.tsx`
  - 其他存在同类结构的博客文章页
- **修复动作**
  - 保留正文区域的主 H1。
  - 将隐藏 SEO 区域中的 H1、H2 改为普通容器标签，例如 `div`、`span`，或直接移除该隐藏标题结构。
  - 不修改原有文字内容，只调整标签语义。
- **预期收益**
  - 保持每页唯一主 H1。
  - 让页面语义结构更清晰，减少搜索引擎解析噪音。

### 4. 重复工具页的 canonical / noindex 规则统一

- **问题**
  - 当前存在较多 `-tool`、`-converter`、`-embed`、近义工具页，可能造成重复内容竞争。
- **涉及文件**
  - `src/app/tools/**/page.tsx`
  - `src/app/layout.tsx`
  - 如后续抽配置，则涉及 `src/data/toolRegistry.ts`
- **修复动作**
  - 明确每组同义工具页的主版本。
  - 主版本页面：允许索引并设置自引用 canonical。
  - 非主版本页面：保留访问，但 canonical 指向主版本；必要时增加 `noindex,follow`。
  - embed 模式页面不作为独立索引目标。
- **预期收益**
  - 降低关键词内耗。
  - 将抓取和排名信号集中到主页面。

## P1-本周执行

### 1. sitemap 从手写列表改为统一生成逻辑

- **问题**
  - 当前 `sitemap` 为手写维护，容易漏页、错页、保留非主页面。
- **涉及文件**
  - `src/app/sitemap.ts`
  - 如抽配置，涉及 `src/data/toolRegistry.ts`
- **修复动作**
  - 将 `sitemap` 改为基于统一页面配置生成。
  - 仅纳入真正需要索引的主页面：
    - 首页
    - 工具主页面
    - 博客首页
    - 博客文章页
    - 核心 `color-shades` 页面
  - 排除 embed 页、重复页、低价值别名页。
- **预期收益**
  - 提升抓取效率。
  - 让搜索引擎更稳定地识别站点主索引页面。

### 2. 首页补充 `WebSite` 结构化数据

- **问题**
  - 首页缺少基础站点结构化数据，搜索引擎对站点整体语义理解较弱。
- **涉及文件**
  - `src/app/layout.tsx`
  - `src/app/page.tsx`
- **修复动作**
  - 为首页增加 `WebSite` 类型的 JSON-LD。
  - 复用现有 title、description、站点 URL，不新增文案。
- **预期收益**
  - 增强站点层语义信号。
  - 提高首页在搜索引擎中的可理解性。

### 3. 工具页补充 `WebApplication` / `SoftwareApplication` 结构化数据

- **问题**
  - 工具页缺少工具类结构化数据，页面能力难以被明确识别。
- **涉及文件**
  - `src/app/tools/**/page.tsx`
  - 可选抽公共 schema 工具文件
- **修复动作**
  - 为核心工具页补充 `WebApplication` 或 `SoftwareApplication` JSON-LD。
  - 先覆盖流量和业务价值更高的工具页，再逐步扩展。
  - 使用现有页面标题、描述、URL，不改文案。
- **预期收益**
  - 强化“在线工具”页面类型识别。
  - 提升工具页的主题一致性与可理解性。

### 4. 博客页补充 `Article` / `BlogPosting` 结构化数据

- **问题**
  - 博客正文页缺少文章类结构化数据，文章页的发布日期、标题、作者等信号未被显式表达。
- **涉及文件**
  - `src/app/blog/*/page.tsx`
  - 可选抽公共 schema 工具文件
- **修复动作**
  - 为博客文章页增加 `Article` 或 `BlogPosting` JSON-LD。
  - 直接复用已有标题、日期、描述、分类，不新增文案。
- **预期收益**
  - 强化文章页语义。
  - 提升文章页在搜索引擎中的可解释性。

### 5. FAQ 区块补充 `FAQPage` 结构化数据

- **问题**
  - 现有 FAQ 内容虽已存在，但未以结构化方式提供给搜索引擎。
- **涉及文件**
  - `src/app/page.tsx`
  - 各工具落地页中包含 FAQ 的页面
- **修复动作**
  - 为已有 FAQ 区块输出 `FAQPage` JSON-LD。
  - 问题和答案直接复用当前页面已有文本，不改文案。
- **预期收益**
  - 将现有 FAQ 内容转换为明确的结构化信号。
  - 提升页面主题覆盖完整性。

### 6. Breadcrumb 结构化数据补充

- **问题**
  - 当前页面层级关系主要依赖导航与链接，缺少显式面包屑结构化数据。
- **涉及文件**
  - `src/app/tools/**/page.tsx`
  - `src/app/blog/*/page.tsx`
  - `src/app/color-shades/**/page.tsx`
- **修复动作**
  - 为工具页、博客页、色阶页补充 `BreadcrumbList` JSON-LD。
  - 路径使用现有导航层级和 URL，不新增新层级名称。
- **预期收益**
  - 强化站点层级结构表达。
  - 改善搜索引擎对页面上下文的理解。

## P2-后续整理

### 1. `toolRegistry` 统一管理页面索引状态

- **问题**
  - 当前工具列表、索引策略、导航、sitemap 可能分散维护，后续容易出现规则不一致。
- **涉及文件**
  - `src/app/tools/page.tsx`
  - `src/components/Navigation.tsx`
  - 新增建议：`src/data/toolRegistry.ts`
- **修复动作**
  - 抽离统一的工具配置源。
  - 在配置中记录：主页面、是否可索引、是否进 sitemap、是否为别名页等状态。
- **预期收益**
  - 让 SEO 规则集中管理。
  - 降低未来新增工具后的维护成本。

### 2. 首页 / 工具页 / 博客页内部链接增强

- **问题**
  - 当前站内链接虽存在，但主题聚类和权重传递仍不够系统。
- **涉及文件**
  - `src/app/page.tsx`
  - `src/app/tools/**/page.tsx`
  - `src/app/blog/*/page.tsx`
- **修复动作**
  - 在相关页面增加“Related Tools”“Related Articles”等模块。
  - 优先使用现有工具名和文章标题作为锚文本，不新写文案。
- **预期收益**
  - 提升抓取深度和权重分发效率。
  - 强化页面主题聚类。

### 3. 多语言路由 hreflang 补充

- **问题**
  - 部分多语言页面已存在，但语言版本替代关系未明确表达。
- **涉及文件**
  - `src/app/tools/en/**`
  - `src/app/tools/es/**`
  - `src/app/tools/pt/**`
  - 对应 metadata 配置文件
- **修复动作**
  - 为对应语言版本页面补充 `hreflang`。
  - 明确自引用语言版本与其他语言替代关系。
- **预期收益**
  - 减少不同语言页面之间的误判与竞争。
  - 提升多语言索引准确性。

### 4. embed 页与主页面的索引边界梳理

- **问题**
  - 当前部分页面支持嵌入，若无统一策略，可能产生低价值重复索引。
- **涉及文件**
  - `src/middleware.ts`
  - 支持 embed 的工具页
- **修复动作**
  - 明确 embed 模式是否参与索引。
  - 优先将 embed 页视为功能承载页，而非独立 SEO 目标页。
  - 结合 canonical / robots 元策略统一处理。
- **预期收益**
  - 降低重复内容风险。
  - 让抓取资源集中在主页面。

### 5. 字体加载优化

- **问题**
  - 当前全局字体加载较多，部分字体只服务于少量功能页面，可能影响首屏性能。
- **涉及文件**
  - `src/app/layout.tsx`
  - 个别依赖特殊字体的工具页
- **修复动作**
  - 将少数页面专用字体从全局加载改为按页面或按组件加载。
  - 保留真正全站通用字体在全局层。
- **预期收益**
  - 降低首页与普通页面的资源负担。
  - 改善 LCP 和整体加载体验。

### 6. 全局脚本加载优化

- **问题**
  - 当前全局层同时加载统计、录屏、广告等第三方脚本，可能影响首屏与交互表现。
- **涉及文件**
  - `src/app/layout.tsx`
- **修复动作**
  - 审查全局脚本的加载时机与必要性。
  - 尽可能延后非关键脚本，避免过早占用主线程。
- **预期收益**
  - 改善交互响应速度。
  - 减少第三方脚本对页面体验的负面影响。

### 7. client / server component 边界收缩

- **问题**
  - 部分页面使用客户端组件范围过大，导致额外 JS 体积和渲染负担。
- **涉及文件**
  - `src/app/page.tsx`
  - `src/app/tools/page.tsx`
  - 其他工具页入口
- **修复动作**
  - 将静态内容尽量转为 server component。
  - 仅保留真正需要交互的局部为 client component。
- **预期收益**
  - 降低客户端 JS 负担。
  - 改善性能与可抓取性。

### 8. 广告导致的 CLS 控制

- **问题**
  - 广告位若未预留稳定尺寸，容易造成布局偏移。
- **涉及文件**
  - 含广告位的工具页与内容页
  - 广告相关组件
- **修复动作**
  - 为广告容器预留固定或可预测尺寸。
  - 减少广告异步加载后引发的布局跳动。
- **预期收益**
  - 改善 CLS。
  - 提升整体页面体验和移动端稳定性。
